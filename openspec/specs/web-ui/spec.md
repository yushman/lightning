## Requirements
### Requirement: Server-rendered HTML without JS build chain
All UI screens SHALL be server-rendered HTML generated by the server binary, with no JavaScript build step and no external asset pipeline. User-provided strings SHALL be HTML-escaped.

#### Scenario: Page renders standalone
- **WHEN** any UI page is fetched with a plain HTTP GET
- **THEN** the response is complete HTML requiring no client-side build artifacts

### Requirement: Flaky tests list screen
The server SHALL serve a screen at `/` listing flaky tests ordered by score descending, showing per test: class and name, score, and a trend of recent verdicts. Each row SHALL link to the test history screen.

#### Scenario: Flaky test listed with trend
- **WHEN** a flaky test exists and `/` is fetched
- **THEN** the page shows the test with its score and recent verdict trend

### Requirement: Test history screen
The server SHALL serve a screen at `/tests/{id}` showing a test's verdict history across runs: per run its verdict, SHA, branch, time, and a link to the run summary.

#### Scenario: History shows same-SHA flip
- **WHEN** a test has a fail and a pass on the same SHA and its history page is fetched
- **THEN** both verdicts appear with that SHA

### Requirement: Run summary screen
The server SHALL serve a screen at `/runs/{id}` summarizing a run: SHA, branch, CI link when present, verdict counts, and the list of failed and mixed tests linking to their history.

#### Scenario: Run summary lists failures
- **WHEN** a run with failing tests is fetched at `/runs/{id}`
- **THEN** the page shows pass/fail counts and each failing test

### Requirement: Builds list screen
The server SHALL serve a screen at `/builds` listing recent builds newest first, showing per build: time, branch, short SHA, outcome, requested tasks, total duration, and cache avoidance ratio (`up-to-date` + `from-cache` tasks over all tasks). Each row SHALL link to the build detail screen.

#### Scenario: Build listed with avoidance ratio
- **WHEN** a build with cached and executed tasks exists and `/builds` is fetched
- **THEN** the page shows the build with its duration and avoidance ratio linking to its detail page

### Requirement: Build detail screen
The server SHALL serve a screen at `/builds/{id}` showing a build's metadata (SHA, branch, outcome, Gradle/JDK versions, CI link when present), configuration versus execution time, a task outcome breakdown with counts per outcome, the build's cache hit rate (`from-cache` over `from-cache` + `success` + `failed` tasks), and the slowest tasks with their outcomes and durations.

#### Scenario: Slow tasks visible
- **WHEN** a build's detail page is fetched
- **THEN** the page shows configuration and total times, outcome counts, and the slowest tasks ordered by duration

#### Scenario: Build hit rate visible
- **WHEN** a build with `from-cache` and executed tasks is fetched at `/builds/{id}`
- **THEN** the page shows the build's cache hit rate

### Requirement: Branch trend screen
The server SHALL serve a screen at `/trends` aggregating recent builds per branch: build count, median total duration of successful builds, and median cache avoidance ratio.

#### Scenario: Branch medians shown
- **WHEN** several builds exist on a branch and `/trends` is fetched
- **THEN** the page shows that branch with its build count and median duration

### Requirement: Section navigation
Every UI page SHALL include navigation links reaching the flaky tests list, the builds list, the trends screen, and the cache analytics screen.

#### Scenario: Builds reachable from flaky radar
- **WHEN** the flaky list at `/` is fetched
- **THEN** the page contains links to `/builds`, `/trends`, and `/cache`

### Requirement: Cache analytics screen
The server SHALL serve a screen at `/cache` showing: cache storage statistics (entry count, total size used, configured size cap, artifact size limit, retention days), the top stored artifacts by hit count (key, size, hits, created, last accessed), the overall task cache hit rate from telemetry (`from-cache` over `from-cache` + `success` + `failed` task executions) with a note that the denominator includes non-cacheable tasks, and a never-cached table: task paths executed at least 3 times across the recent builds window with no `from-cache` or `up-to-date` outcome, ordered by total execution time, with a note that the heuristic cannot distinguish uncacheable tasks from tasks whose inputs always change.

#### Scenario: Storage stats and hit rate shown
- **WHEN** cache entries and telemetry builds exist and `/cache` is fetched
- **THEN** the page shows entry count, total size, top artifacts with hit counts, and the overall hit rate

#### Scenario: Never-cached task listed
- **WHEN** a task path was executed in at least 3 recent builds and never resulted in `from-cache` or `up-to-date`
- **THEN** the page lists that path with its execution count and total execution time
